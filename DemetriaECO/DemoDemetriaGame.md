# DemoDemetriaGame — Project Memory

## ใครคือใคร
- ผู้ใช้: **พี่แยม** (YamuJung) — เจ้าของโปรเจกต์
- ผู้ช่วย: ให้เรียกตัวเองว่า **น้องเดฟ** เสมอในโปรเจกต์นี้
- โทน: เป็นกันเอง ตอบตรงประเด็น กระชับ ไม่ยืดเยื้อ

## บริบทโปรเจกต์
กำลังพัฒนาเกม isometric object-placement/simulation เพื่อ **วางจำหน่ายบน Steam (Windows + Mac)** โดยตัวเกมจริงจะสร้างด้วย **Unity**

ก่อนหน้านี้ทำ **เดโม่ต้นแบบ (prototype)** ด้วย HTML5 Canvas + vanilla JavaScript ล้วน (ไม่มี engine, ไม่มี asset ภาพ) ชื่อไฟล์ `isometric_game.html` เพื่อทดลองออกแบบ "กฎของระบบเกม" ให้ลงตัวก่อน แล้วค่อยย้ายไปเขียนจริงใน Unity — เดโม่นี้ไม่ได้มีแผนจะ ship ขึ้น Steam ตรงๆ มีหน้าที่เป็นแค่เครื่องมือพิสูจน์แนวคิดเท่านั้น

### ระบบพื้นฐาน (v1.0–v2.6)
- **Isometric projection พื้นฐาน**: กระดาน grid↔screen แบบ diamond 2:1 — **ขนาดกระดานปรับได้เอง (v3.89)** ไม่ใช่ 12×12 ตายตัวอีกต่อไป (พิมพ์กว้าง/ยาวใหม่แล้วกดปรับ ช่วง 4-40 ต่อด้าน) ขยายได้เสมอ แต่ **ย่อไม่ได้ถ้ามีของ/พื้นผิว/ตำแหน่งผู้เล่นอยู่นอกขอบใหม่** (บล็อกทั้งหมด ไม่ใช่แค่เตือน — ป้องกันข้อมูลหายแบบเงียบๆ) ขนาดกระดานต้องบันทึกไปกับไฟล์แผนที่ด้วยเสมอ ไม่งั้น import กลับมาแล้วเพี้ยน — ใน Unity แปลว่าขนาด Tilemap/เขตแดนที่เดินได้ควรเป็นค่าที่ปรับได้และ serialize ไปกับ save data เดียวกับ layout ไม่ hardcode
- **Build mode vs Play mode**: Build mode ใช้แก้แผนที่ (มีเส้น Grid, ผู้เล่นซ่อน/หยุดนิ่ง) ส่วน Play mode ใช้เดินเล่น (ไม่มีเส้น Grid, ผู้เล่นโชว์ตัวและเดินได้ด้วย WASD/ลูกศร)
- **การวางวัตถุละเอียดครึ่งช่อง (half-tile snapping)** สำหรับวัตถุ footprint เต็มช่อง (กล่อง กำแพง ต้นไม้ ก้อนหิน ถัง โต๊ะ ประตู หีบ คันโยก ฯลฯ)
- **การวางวัตถุ footprint เล็ก (เช่น แจกัน)**: สแนปเข้า 1 ใน 4 ช่องย่อย (quadrant) ของแต่ละช่องใหญ่ ตามเส้นประ sub-grid
- **Hit-testing แบบ silhouette**: ชี้/คลิก/หมุน/ลากวัตถุ เช็คจากรูปทรงจริงที่มองเห็นบนจอ (หลังคา+ผนังด้านที่เห็น) ไม่ใช่แค่ช่อง Grid พื้นราบ — ของที่อยู่เลเยอร์นอกสุด (เช่น แจกันบนโต๊ะ) ต้องเช็คก่อนของที่อยู่เลเยอร์ล่าง (โต๊ะ)
- **Elevation-aware target resolution**: เวลาวางของบนพื้นผิวที่ยกสูง (เช่น หน้าโต๊ะ) ต้องเช็ค silhouette ของวัตถุที่ยกสูงนั้นก่อน ไม่ใช้การคำนวณ ground-plane ตรงๆ (ไม่งั้นจุดวางจะเพี้ยนไปตกช่องข้างหลัง)
- **Zoom + Pan**: scroll wheel/ปุ่มซูม center-anchored (คำนวณพิกัดเมาส์กลับเป็นพิกัดโลกต้องคูณ/หารด้วย zoom เสมอ), กลางเมาส์ (middle-click) ลากแพนแผนที่ใน Build mode, กล้อง Play mode ตามผู้เล่นอัตโนมัติแบบ ease smoothing
- **Painter's algorithm depth-sort**: `depth = col + row` ใช้ทั้งลำดับการวาดและตัดสิน "ใครอยู่หน้าใคร" เวลามีวัตถุซ้อนกัน
- **สิ่งกีดขวาง Block (1x1x1)**: วัตถุพื้นฐานเสริมจาก v2.6
- **การชนของผู้เล่น (collision) ต้องเทียบกับขนาดจริงที่มองเห็น ไม่ใช่แค่ปัดเข้าตาราง** (บทเรียนสำคัญจากบั๊กจริง v3.64): เดิมเช็คชนโดยปัดตำแหน่งผู้เล่นเข้าจุดกริดที่ใกล้ที่สุดแล้วเทียบคีย์ตรงๆ ซึ่งดูแลพื้นที่แค่ครึ่งเดียวของขนาดที่วัตถุแสดงผลจริง ทำให้เดินทะลุเข้าเนื้อวัตถุได้ก่อนโดนหยุด — ต้องเช็คแบบ "จุดต่อเนื่องอยู่ในขอบเขตจริงของวัตถุไหม" (point-in-footprint/AABB) แทน ใน Unity ใช้ Collider2D/Rigidbody2D ของจริงแทนได้เลย ปัญหานี้จะไม่เกิด เพราะ physics engine เช็คแบบ continuous อยู่แล้ว
- **การเดินทแยงมุม (diagonal-only input) ต้องเช็คจุดหมายรวมก่อนเสมอ ห้ามแยกแกน X/Y เช็คคนละรอบแบบใช้ค่าตำแหน่งเก่า/ใหม่ปนกัน** (บทเรียนจากบั๊กจริง v3.64 เช่นกัน): เกมนี้ผู้เล่นเดินด้วย WASD ซึ่งเป็นทิศทแยงมุมล้วนๆ ไม่มีปุ่มเดินแกนเดียว ถ้าระบบเช็คแกนนอน-แกนตั้งแยกกันคนละรอบและใช้ตำแหน่งอีกแกนที่ "ยังไม่อัปเดต" ของเฟรมนั้น จะเปิดช่องให้ไถลอ้อมทะลุวัตถุเดี่ยวๆ ไปได้โดยไม่โดนบล็อกเลยสักครั้ง (ตรวจพบและยืนยันด้วยการจำลอง step-by-step จริง) — วิธีแก้ที่ถูกต้อง: เช็คจุดหมายทแยงมุมเต็มๆ ก่อนเสมอ ถ้าติดค่อย fallback ไปเช็คสไลด์ทีละแกนแบบ mutually exclusive (else-if ไม่ใช่ if แยก 2 อัน) เพื่อกันไม่ให้ 2 การขยับที่ "ดูฟรีแยกกัน" รวมกันแล้วกลับไปโดนจุดที่บล็อกไว้ — ใน Unity ถ้าใช้ Rigidbody2D + Collider2D มาตรฐานจะไม่เจอปัญหานี้ (physics engine จัดการ sliding ถูกต้องอยู่แล้ว) แต่ถ้าจะเขียนระบบเช็คกริดเองแบบกำหนดเอง ต้องระวังจุดนี้เป็นพิเศษ

### ระบบวัตถุ/เฟอร์นิเจอร์ (v3.x)
- **ชนิดวัตถุทั้งหมด**: box, wall, wallbox/wallbox5 (กรอบรูปแขวนผนัง), block, tree, rock, barrel, table (โต๊ะใหญ่), tableSmall (โต๊ะเล็ก 1x1), vase (แจกัน), giftBox (กล่องของขวัญ), door, chest, lever, coin, gem, npc, player
- **ระบบ facing/flip 4 ท่า (`FACING_CYCLE_TYPES`)**: npc, table, vase, tableSmall, giftBox — คลิกขวาวนหน้า/หลัง/กลับด้าน (4 สถานะรวมกัน) ผ่าน `state.facing` + `state.flip`, แต่ละชนิดอัปโหลดรูป front/back แยกกันได้
- **ระบบ "topper" ทั่วไปบนโต๊ะใหญ่ (`TABLE_TOPPER_TYPES`)**: เดิมรับแค่แจกัน ขยายเป็น Set รองรับหลายชนิด (แจกัน + กล่องของขวัญ) — โต๊ะ 1 ตัวรับได้ 4 ชิ้นพร้อมกัน แยกอิสระ 4 มุม (บน/ขวา/ล่าง/ซ้าย) ย้ายออก/ย้ายไปโต๊ะอื่นได้ — เพิ่มชนิดใหม่ในอนาคตแค่เพิ่มเข้า Set นี้ ไม่ต้องแก้ทุกจุด
- **วัตถุ footprint เล็กซ้อนกันหลายชั้น (stacking)**: `findFreeLayer` หาเลเยอร์ว่างถัดไปในควอดรันต์เดียวกันอัตโนมัติเมื่อ drop ทับของเดิม (เช่น วางแจกันซ้อนบนโต๊ะเล็ก) ไม่ต้องกด Tab ค้าง
- **Stack ของ footprint เล็กเคลื่อนไหวเป็นก้อนเดียวกัน (v3.57/v3.58)**: ย้าย (drag) หรือหมุน/Flip (คลิกขวา) วัตถุฐานที่มีของซ้อนอยู่ข้างบน (เช่นของวางบนโต๊ะเล็ก) จะพาของที่ซ้อนอยู่ทั้งหมดเคลื่อนไหว/หมุนตามไปด้วยในการกระทำเดียวกันเสมอ — เหมือนเป็น rigid group ชั่วคราว ถ้าย้ายแล้วที่ใหม่ไม่มีที่พอสำหรับทั้งกอง ต้องยกเลิกและคืนกลับที่เดิมทั้งหมด ห้ามวางลงครึ่งเดียวแล้วทำของหาย ส่วนการหยิบ/คลิกที่ "ของบนสุด" ตัวเดียว (ไม่ใช่ฐาน) ยังคงเคลื่อนไหวแค่ตัวนั้นเหมือนเดิม ไม่กระทบของด้านล่าง

### ระบบ Build mode ย่อย: Foundation vs Furniture (v3.77)
- Build mode แบ่งเป็น 2 โหมดย่อยที่สลับได้ (ปุ่ม toggle): **Foundation** (สิ่งกีดขวาง/กำแพง/พื้นผิว/ประตู-หีบ-คันโยก — งานโครงสร้าง) กับ **Furniture** (เฟอร์นิเจอร์/ของเก็บได้/ตัวละคร — งานตกแต่ง) — พาเลทฝั่งซ้ายกรองโชว์ตามโหมดที่เลือก และ**แก้ไข/ลบ/ย้ายของบนแผนที่ได้เฉพาะกลุ่มที่ active อยู่เท่านั้น** ของอีกกลุ่มยังเห็นปกติแต่แตะไม่ติด — กันไม่ให้เผลอกระทบของผิดหมวดตอนโฟกัสทำงานอีกฝั่ง แนวคิดนี้เอาไปใช้กับ Unity editor tool ได้ตรงๆ (จำกัดสิทธิ์แก้ไขตาม layer/category ที่ active)
- **บทเรียนจากบั๊กจริง (v3.77→v3.78)**: ตอน implement โหมดนี้ เขียนโค้ด sync สถานะเริ่มต้นของ UI (DOM) ด้วยการเรียกฟังก์ชัน sync ทันทีที่ตำแหน่งที่แทรกโค้ดใหม่ในไฟล์ — แต่ตำแหน่งนั้นอยู่ **ก่อน** บรรทัดที่ประกาศ `let` ตัวแปรอื่นที่ฟังก์ชันนั้นอ้างถึง (แม้ฟังก์ชันจะถูกเรียกตอน "runtime" ก็ตาม เพราะ JS รันสคริปต์ท็อปเลเวลจากบนลงล่างครั้งเดียวตอนโหลดหน้า ไม่ใช่ event-driven) ทำให้สคริปต์ทั้งไฟล์ throw error และพังตั้งแต่โหลดหน้าเลย ทั้ง Build/Play mode ใช้งานไม่ได้เลยสักปุ่ม — จับได้เพราะพี่แยมลองเล่นจริงแล้วแจ้งว่า "พื้นที่เล่นเกมจะบั๊ก" วิธีแก้: ให้ HTML/CSS เริ่มต้นตรงกับค่า default ของตัวแปรอยู่แล้วตั้งแต่แรก (ไม่ต้องมีโค้ด sync เรียกทันทีตอนท็อปเลเวล) เหมือนที่ระบบ Build/Play เดิมทำอยู่แล้ว — **บทเรียนทั่วไป**: ระวังการเรียกฟังก์ชันที่อ้างตัวแปรอื่นทันทีที่ตำแหน่ง top-level ของสคริปต์ ให้เช็คว่าตัวแปรที่อ้างถึงถูกประกาศไปแล้วจริงๆ ณ จุดนั้น ไม่ใช่แค่ "ประกาศอยู่ในไฟล์เดียวกัน"

### ระบบพื้นผิว (terrain) — ผ่านการลองผิดลองถูกหลายรอบจนได้ดีไซน์สุดท้าย
- **ดีไซน์สุดท้าย (v3.85→v3.88)**: พื้นผิวแต่ละชนิด (หญ้า/ทราย/น้ำ ฯลฯ) เป็น **สีเรียบสีเดียวต่อชนิด ไม่มีลายตารางหมากรุก ไม่มีการผสมสีอัตโนมัติกับเพื่อนบ้านต่างชนิด** — ช่องที่ต่อกันแค่ "ล้นขอบทับกันเล็กน้อย" (~1.5px) เพื่อกันรอยต่อจาง (anti-aliasing seam) ระหว่างช่องเท่านั้น ไม่ใช่เพื่อผสมสี ถ้าอยากได้ลวดลายเปลี่ยนผ่าน (เช่น ขอบหญ้า-ทราย) ให้สร้างเป็น **ชนิดพื้นผิวแยกต่างหากที่วางเองด้วยมือ** (ดูหัวข้อถัดไป) ไม่ใช่ระบบเดาอัตโนมัติจากเพื่อนบ้าน — ลองระบบเดาอัตโนมัติ (neighbor-based auto-blend) มาแล้วในช่วง v3.80 แต่พี่แยมไม่ชอบผลลัพธ์ที่ควบคุมไม่ได้ ขอถอดออกทั้งหมด ใน Unity: ใช้ Tilemap ธรรมดา + RuleTile/manual tile variant ที่ศิลปินวาดเองแทนได้เลย ไม่ต้องเขียนระบบ auto-blend เอง
- **ระบบ "variant kit"**: พื้นผิวชนิดเดียวกัน (เช่น หญ้า) มีได้หลาย instance อิสระจากกัน (หญ้า 1-9) แต่ละอันอัปโหลดรูปคนละรูป ตั้ง ID/ชื่อเองได้แยกกัน — ใช้ตอนอยากได้พื้นผิวแบบเดียวกันแต่หลายลาย/หลายภาพสลับใช้ ไม่ใช่ auto-generate ลาย เป็น pattern ที่ทำซ้ำได้กับวัตถุชนิดอื่นถ้าพี่แยมต้องการ (สร้าง N ชนิดที่ behavior เหมือนกันแต่ asset ต่างกัน แยก type key แต่ละอัน ไม่ผูกกับ type เดิม)
- **UI ที่ยาวเกินไปให้เก็บใน accordion**: ชุดพื้นผิวที่มีหลาย variant (เช่น หญ้า 1-9) ควรพับเก็บใน `<details>` แยกจากรายการหลักที่โชว์ตลอด กันพาเลทยาวเกินจอ
- **การลบ/แก้ไขที่ทำลายข้อมูลได้ ต้องมีท่าทางยืนยันชัดเจนก่อนเสมอ**: เดิมคลิกพื้นที่ว่างเฉยๆ ก็ลบพื้นผิวทันที เผลอคลิกโดนของหายง่าย — เปลี่ยนเป็นต้องกดปุ่ม "เข้าโหมดลบ" ก่อน (ปุ่มติดสี, cursor เปลี่ยน) แล้วชี้เมาส์จะขึ้นกรอบไฮไลต์ (สีเหลือง) บอกล่วงหน้าว่ากำลังจะลบอันไหน ค่อยคลิกยืนยัน — ใช้ pattern เดียวกับระบบทาสีกำแพง (arm mode ด้วยปุ่ม/สวอตช์ → คลิกเป้าหมายจริง) แนวคิด "arm-then-confirm พร้อม preview ไฮไลต์" นี้ใช้กับ editor tool ที่มีการกระทำทำลายล้างได้ (delete/erase) ได้ทุกที่ ไม่ใช่แค่เกมนี้

### ระบบพื้นผิวเล็ก (terrain decal) — "หญ้า/ทราย/น้ำ (เล็ก)" (v3.100–v3.110)
ต่อยอดจากระบบพื้นผิวใหญ่ด้านบน — เพิ่มเลเยอร์พื้นผิวขนาดเล็กที่วางซ้อนบนพื้นใหญ่ได้ (คนละเลเยอร์ ลบ/ทาสีแยกจากกัน) ผ่านการออกแบบผิดหลายรอบกว่าจะนิ่ง:
- **ดีไซน์สุดท้าย**: สแนปเข้า **1 ใน 4 ควอดรันต์** ของช่องตามตำแหน่งเมาส์ (ใช้ระบบควอดรันต์เดียวกับที่แจกัน/โต๊ะเล็กใช้อยู่แล้ว — `getTileQuadrant`/`smallKey`) แล้ว**เต็มพอดีควอดรันต์นั้นเป๊ะๆ ไม่มีขอบเว้น** วางได้สูงสุด **4 ชิ้นอิสระต่อ 1 ช่องใหญ่** (คนละมุม คนละชนิดก็ได้) — ก่อนจะนิ่งที่ดีไซน์นี้ เคยลองมาแล้ว 2 แบบที่พี่แยมปฏิเสธ: (1) ของชิ้นเดียวลอยกึ่งกลางช่องขนาด 1/4 เชิงเส้น (2) ของชิ้นเดียวขนาดเกือบเต็มช่อง (~94%) — ทั้งสองแบบผิดเพราะเข้าใจคำว่า "1x1 ช่องเล็ก" ผิด (พี่แยมหมายถึงควอดรันต์ ไม่ใช่ของชิ้นเล็กวางกลางช่องทั้งแผ่น) กว่าจะเข้าใจตรงกันต้องรอพี่แยมส่งภาพวงกลมชี้ตำแหน่งที่ต้องการมาให้ดูตรงๆ — **บทเรียน**: คำอธิบายด้วยคำพูดล้วนๆ (เช่น "ขนาด 1x1") อาจตีความกำกวมได้หลายแบบ ถ้าคุยกันแล้วยัง งง ให้ขอภาพตัวอย่าง/ให้วงจุดที่ต้องการชี้ตรงๆ ดีกว่าเดาต่อ
- **เหตุผลที่ต้องมีเลเยอร์แยก ไม่ใช้พื้นผิวใหญ่ตรงๆ**: ต้องการให้ลบ/ย้าย/ทาสีของเล็กได้โดยไม่กระทบพื้นใหญ่ด้านล่าง และวางซ้อนพื้นใหญ่ที่มีอยู่แล้วได้ — ใน Unity คือ Tilemap ชั้นที่ 2 (หรือ sprite renderer แยก) เรียงอยู่เหนือ Tilemap พื้นฐาน ไม่ใช่ tile ชนิดเดียวกันที่สลับสีในช่องเดิม

**บทเรียนเรื่องรอยต่อ (seam) — สำคัญมากถ้าจะเขียนระบบวาดแบบ manual-bleed เองที่ไหนอีก (ไม่จำเป็นถ้าใช้ Unity Tilemap ตรงๆ เพราะ built-in renderer จัดการ pixel-perfect adjacent tiles ให้อยู่แล้ว ไม่ต้องทำ bleed math เอง):**
- **scale จาก "จุดศูนย์กลางร่วม" ไม่ใช่จุดศูนย์กลางของตัวเอง** ปิดรอยต่อที่จุดยอดร่วมกันได้แน่นอน (พิสูจน์ทางคณิตศาสตร์ ไม่ใช่แค่ทดลอง) ส่วน scale จากศูนย์กลางตัวเองใช้ได้เฉพาะกรณี "รูปทรงเดียวกันแค่เลื่อนตำแหน่ง" (เช่น 2 ช่องเต็มข้างกัน) แต่ใช้ไม่ได้กับกรณี "รูปทรงต่างกันมาบรรจบที่จุดเดียว" (เช่น 4 ควอดรันต์รอบจุดศูนย์กลางช่อง หรือ 4 ช่องเต็มมาเจอกันที่มุม) — ถ้าเจอมุมที่มี object หลายชิ้นบรรจบกัน ต้องเช็คว่า scale จากจุดไหน ไม่ใช่แค่ "scale จากตัวเอง" เป็น default
- **ของสีเดียวกันที่ชนกัน ต้องรวมเป็นคำสั่งวาด (fill call) เดียว ไม่แยกวาดทีละชิ้น** แม้เรขาคณิตจะตรงกันเป๊ะแล้วก็ตาม — เพราะการ anti-alias ของแต่ละคำสั่งวาดแยกกันจะเบลนด์ขอบตัวเองอิสระจากกัน ทำให้เกิดรอยสว่างผิดปกติ (double-blend) ตรงรอยต่อได้แม้จะเป็นสีเดียวกัน วิธีแก้คือ group ตามสี/material แล้ว fill ทีเดียวทั้งกลุ่ม
- **ของสีต่างกันที่ชนกันและ "ทับกันจริง" (overlap) โดยตั้งใจ (เพื่อกันรอยจาง) จะมีปัญหาลำดับการวาด**: ถ้า group ตามสีแล้ววาดทีละกลุ่ม สีที่ถูกวาด "ทีหลัง" จะชนะพื้นที่ทับซ้อนเสมอ — ถ้าลำดับกลุ่มขึ้นกับ "ลำดับที่ถูกวางลงกระดานครั้งแรก" (เช่น ใช้ insertion order ของ Map) จะกลายเป็นว่า "สีที่วางทีหลังบนกระดานทั้งหมด ชนะทุกจุดที่มันไปชนกับสีอื่นเสมอ" — บั๊กนี้วัดได้จริง (~4% พื้นที่ไม่เท่ากัน) ไม่ใช่แค่ภาพลวงตาสี วิธีแก้ที่ดีที่สุด: ถ้า scale จากจุดศูนย์กลางร่วม (ข้อแรก) ทำให้ขอบที่ใช้ร่วมกันของ 2 รูปตรงกันแบบ "จุดเดียวกันเป๊ะ" (ไม่ใช่แค่ใกล้กัน) อยู่แล้ว มักไม่ต้องมี "ระยะทับซ้อนเสริม" เพิ่มเลย (ลองปิดค่านี้เป็น 0 แล้ววัดพิสูจน์ก่อนใส่กลับ)
- **เทคนิคตรวจรอยต่อที่แม่นกว่า "นับพิกเซลสีตรงกัน"**: การนับพิกเซลที่ตรงสีเป๊ะ (exact color match) เพื่อวัดพื้นที่ อาจให้ผลคลาดเคลื่อนถ้าสี 2 ฝั่งมี contrast ต่างกันมาก (เช่น สีสว่างจัดเทียบสีเข้ม) เพราะโซนพิกเซลที่ blend/anti-alias จะถูกตัดออกจากทั้ง 2 ฝั่งแบบไม่สมมาตร — วิธีที่แม่นกว่าคือ **วัดตำแหน่งเส้นขอบจริงตรงๆ** (sample พิกเซลไล่ตั้งฉากกับเส้นขอบทฤษฎี ดูว่าจุดเปลี่ยนสีอยู่ตรงกลางพอดีไหม) ควบคู่กับซูมภาพจริงแบบ 20-25 เท่า (ระดับที่เคยจับบั๊กรอยต่อจริงได้มาก่อน)
- **มุมที่ 4 ช่อง (พื้นใหญ่ ไม่ใช่ควอดรันต์) มาบรรจบกันก็มีบั๊กรอยต่อแบบเดียวกัน แต่ **ยังไม่ได้แก้** ในเดโม่ปัจจุบัน — เคยออกแบบ+ทำ+ทดสอบผ่านแล้ว (เปลี่ยนพื้นใหญ่จากสี่เหลี่ยม 4 มุมง่ายๆ เป็นรูป 8 มุม: 4 มุมจริงตรึงตำแหน่งไม่ขยับ + 4 กึ่งกลางขอบดันออกแบบเดิม) แต่พี่แยมขอถอดกลับเพราะไม่ต้องการการเปลี่ยนแปลงนี้ตอนนั้น — **ถ้ากลับมาแก้อีกครั้ง**: แนวทางที่พิสูจน์แล้วว่าใช้ได้อยู่ใน `changelog.html` entry v3.108 (ก่อนถูก revert ใน v3.109) เป็นจุดเริ่มต้นที่ดี

**บทเรียนกระบวนการทำงาน**: ในเซสชันเดียวกันนี้ เผลอข้ามขั้นตอน "ส่ง outline ก่อนแล้วรอยืนยัน" ไปครั้งหนึ่ง (ตอนแก้บั๊กรอยต่อสี v3.110 — คิดว่าเป็น "แค่บั๊กฟิกซ์ที่มีหลักฐานพิสูจน์แน่นแล้ว" เลยลงมือแก้ตรงๆ) พี่แยมเตือนกลับมาทันที ("Outline มาให้ก่อนจ้า") — **กฎเดิมในหัวข้อ "แนวทางทำงานร่วมกัน" ด้านล่างไม่มีข้อยกเว้น แม้จะเป็นบั๊กฟิกซ์ที่วัดพิสูจน์มาแน่นหนาแค่ไหนก็ตาม ต้องส่ง outline ก่อนแตะโค้ดเสมอ**

### ระบบรูปภาพที่อัปโหลดเอง (custom image)
- แต่ละชนิดวัตถุมีกรอบขนาดสูงสุด (Max Width/Height) แยกอิสระ ปรับได้เองผ่านพาเนล "ขนาดรูป" ใน build mode — ถ้าไม่อัปโหลดรูป จะ fallback ไปใช้รูปทรง procedural vector ที่วาดเอง (drawBox, drawTable, drawVase, drawTableSmall, drawGiftBox, drawFigure ฯลฯ)
- **Offset พื้น (Ground Offset) ปรับได้ต่อชนิด**: NPC/Player, โต๊ะเล็ก, กล่องของขวัญ มีค่า offset (px) แยกกัน ปรับ -200 ถึง 200 เผื่อรูปที่อัปโหลดมีระยะขอบไม่ตรงกับสมมติฐานของ engine (default = ค่าคงเดิมของระบบ ไม่กระทบของที่วางไว้แล้วจนกว่าจะปรับเอง)
- **แยก "ขนาดที่แสดงผล" ออกจาก "พื้นที่ hit-test"** (บทเรียนสำคัญจากบั๊กจริง v3.49→v3.54): วัตถุ footprint เล็ก (tableSmall, giftBox) ต้อง "ตรึง" ขนาดกรอบเหลืองที่แสดงผลให้เล็กเท่า 1x1 เสมอ (ไม่งั้นดูใหญ่ผิดปกติเทียบของข้างเคียง) แต่พื้นที่ตรวจจับคลิก/hover ต้องอิงขนาดรูปจริงที่อัปโหลด (ไม่งั้นคลิกโดนแค่มุมเดียวใกล้ฐาน) — แก้ด้วย parameter `forHitTest` แยก 2 โหมดในฟังก์ชันคำนวณ silhouette เดียวกัน โดยฟังก์ชันที่ "วาด" ยังเรียกแบบเดิม (เล็ก) ส่วนฟังก์ชันที่ "ตรวจจับคลิก" เรียกแบบใหม่ (ขนาดจริง) — แนวคิดนี้ใช้ได้กับ engine ไหนก็ตามที่แยก visual bounds กับ collision bounds ออกจากกันได้
- **บทเรียนต่อยอด (v3.59→v3.63)**: การให้ hit-test "อิงขนาดรูปจริง" ตามข้อบนดูดีในทฤษฎี แต่ในทางปฏิบัติไล่แก้ edge case ไม่จบ (พื้นที่โปร่งใสในไฟล์รูป, ค่า Ground Offset ที่ไม่ถูกชดเชย, กรอบขนาดสูงสุดที่ตั้งไว้เองก็ใหญ่จริงตามที่ตั้ง ฯลฯ) สุดท้ายตัดสินใจ **ล็อคพื้นที่ hit-test ของวัตถุ footprint เล็ก (tableSmall, giftBox) ให้ตรงกับ footprint ตรรกะเดิม (1x1 ควอดรันต์) ตายตัว ไม่อิงรูปที่อัปโหลดเลยแม้แต่น้อย** (ทั้งขนาด สัดส่วน หรือ Offset) แลกกับข้อจำกัดที่รูปสูงเกินกรอบจะคลิกส่วนที่โผล่พ้นไม่ติด — ใน Unity แปลว่า: **ใช้ Collider2D ขนาดคงที่ตาม prefab เดิม ไม่ต้อง resize collider ตามขนาด sprite/texture ที่สลับเข้ามาใหม่** สำหรับวัตถุตกแต่งเล็กๆ พวกนี้ ง่ายกว่าและคาดเดาได้แน่นอนกว่าการพยายามคำนวณ collider ให้ตรงกับภาพเป๊ะๆ ทุกครั้ง

### ระบบบันทึกข้อมูล
- **บันทึกรูปที่อัปโหลดถาวรด้วย IndexedDB**: อัปโหลดรูปครั้งเดียว ไม่ต้องอัปโหลดซ้ำทุกครั้งที่เปิดเกมใหม่ — เลือก IndexedDB แทน localStorage เพราะรองรับข้อมูลขนาดใหญ่กว่ามาก และเก็บฝั่ง browser ผู้ใช้เท่านั้น (ไม่กระทบ repo ถ้า host บน GitHub Pages) มี migration อัตโนมัติจาก localStorage เวอร์ชันเก่า
- **Export/Import รูปที่บันทึกไว้**: ปุ่ม Export รวมรูปทั้งหมดใน IndexedDB เป็นไฟล์ .json เดียวดาวน์โหลดได้ ใช้สำรองข้อมูลหรือย้ายเครื่อง/เบราว์เซอร์ Import อ่านไฟล์กลับเข้า ถามยืนยันก่อน แล้วเขียนทับเฉพาะชนิดที่ตรงกัน (merge ไม่ใช่ล้างทั้งหมดแบบ import แผนที่)
- **Export/Import แผนที่ (.json)**: แยกจากระบบรูปภาพข้างต้น เก็บเฉพาะ layout วัตถุบนกระดาน
- **ค่าตั้งค่าขนาด/Offset ของรูปก็ต้องบันทึกถาวรด้วย ไม่ใช่แค่ตัวรูป (v3.61)**: ตอนแรกบันทึกแค่ไฟล์รูปที่อัปโหลดผ่าน IndexedDB แต่ค่าตัวเลข (กรอบขนาดสูงสุด, Ground Offset) ที่พี่แยมปรับเองยังอยู่แค่ในตัวแปรระหว่างรัน โหลดหน้าใหม่แล้วรีเซ็ตกลับ default ทุกครั้ง — ต้องบันทึกค่าปรับแต่งพวกนี้ไปพร้อมกับ asset ด้วยเสมอ ใน Unity คือของที่ควรเก็บใน save data/ScriptableObject เดียวกับ reference asset ไม่ใช่แยกกันคนละที่

## ข้อจำกัดของเดโม่ (เหตุผลที่ต้องย้ายไป Unity)
Canvas 2D วิ่งบน CPU ล้วน ไม่มีระบบ save/load, ไม่มี multiplayer, เป็น 2.5D ไม่ใช่ 3D จริง, ไม่รองรับ touch, ไม่มีเสียง — Unity จะแก้ข้อจำกัดเหล่านี้ทั้งหมด และมี built-in tilemap/physics/hit-test ที่ช่วยลดงานเขียนมือ

## แนวทางทำงานร่วมกัน
- ก่อนแก้ไข/เพิ่มฟีเจอร์ที่มีผลต่อ visual หรือ behavior ชัดเจน ให้ทำ preview/outline (มักเป็น SVG diagram) ให้พี่แยมดูก่อนเสมอ แล้วรอคำยืนยัน (เช่น "โอเคจ้า") ก่อน implement จริง — **ไม่มีข้อยกเว้น แม้จะเป็นบั๊กฟิกซ์ที่มั่นใจมาก/มีหลักฐานทดสอบพิสูจน์แน่นหนาแล้วก็ตาม** (เคยเผลอข้ามขั้นตอนนี้ตอนแก้บั๊กรอยต่อสีใน v3.110 ด้วยเหตุผลว่า "แค่บั๊กฟิกซ์" พี่แยมเตือนกลับทันที — ความมั่นใจในผลลัพธ์ ไม่ใช่เหตุผลให้ข้ามการขอความเห็นชอบเรื่องแนวทางก่อนแตะโค้ด)
- ทุกครั้งที่แก้ไข ให้ตรวจสอบก่อนส่งงานเสมอ — สำหรับเดโม่ HTML ใช้ syntax check / div-balance check / unit test แยกฟังก์ชัน / jsdom full-page smoke test; สำหรับ Unity ให้ปรับเป็นเครื่องมือของ Unity เอง (compile ผ่านไม่มี error ใน console, Unity MCP's run_tests ถ้ามี)
  - **หมายเหตุสภาพแวดล้อม (ณ 2026-07-09)**: `jsdom` ในเครื่องนี้รันไม่ได้ (ต้องการ Node ^20.19.0 ขึ้นไป แต่เครื่องมี Node 20.15.1) จนกว่าจะอัปเดต Node ให้ข้าม jsdom full-page smoke test ไปก่อน แล้วใช้วิธี "คัดลอก logic ฟังก์ชันล้วนๆ (pure function ไม่พึ่ง DOM/canvas) มาเขียนเป็น standalone Node script แยก แล้วรันจำลองด้วยข้อมูลจริง/edge case จริง" แทน — วิธีนี้พิสูจน์ตัวเองมีประโยชน์มากแม้กระทั่งจับบั๊กจริงได้ (เช่น bug การชนแบบทแยงมุมใน v3.64 ที่ไล่หาสาเหตุด้วยการจำลอง step-by-step จริงจนเจอ ไม่ใช่แค่เดา)
- อัปเดต changelog ทุกครั้งที่มีการเปลี่ยนแปลง (บันทึกทั้งเหตุผล/root cause ของบั๊กที่แก้ ไม่ใช่แค่ว่าทำอะไร — ช่วยตามงานย้อนหลังได้ง่ายขึ้นมาก)
- เมื่อย้ายไปเขียนใน Claude Code: ให้ยังคงเรียกตัวเองว่า "น้องเดฟ" และเรียกผู้ใช้ว่า "พี่แยม" เหมือนเดิม อ่าน skill `yam-nong-dev` (SKILL.md) ประกอบไฟล์นี้เสมอเพื่อโหลด persona + บริบทระบบเกมทั้งหมด
